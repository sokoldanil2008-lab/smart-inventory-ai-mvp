# smart-inventory-ai-mvp<br>
SmartInventory AI<br>
1.Пітч<br>
SmartInventory AI вирішує проблему хаотичного обліку залишків та неочікуваного дефіциту товарів у малого бізнесу. Наша платформа допомагає власникам магазинів та складів автоматизувати рутину за допомогою ШІ, який прогнозує потребу в закупках на основі історії продажів. У результаті користувач отримує оптимізований капітал (гроші не "заморожені" в зайвому товарі) та впевненість, що ходові позиції завжди в наявності.<br>
2.MVP<br>
Ключовий сценарій, без якого продукт не має сенсу: автоматичне сповіщення про критичний залишок.<br>
Користувач завантажує список товарів з поточною кількістю.<br>
Система фіксує факт продажу/списання.<br>
ШІ аналізує швидкість збуту і, коли товару залишається на 3 дні роботи, надсилає сповіщення: "Час замовити Товар Х, орієнтовна кількість — 50 шт".<br>
3.Карта стейкхолдерів (Stakeholder Map)<br>
Отримувачі цінності: Власники малого бізнесу та менеджери зі закупівель, які позбуваються страху забути про замовлення.<br>
Оцінювачі результату: Кінцеві клієнти бізнесу (завжди знаходять товар на полиці) та фінансові контролери (бачать зменшення збитків від простою).<br>
Впливові сторони: Розробники (обмежені складністю інтеграції ШІ-моделей) та постачальники даних (API облікових систем), які диктують швидкість оновлення інформації.<br>
4. Мета першої ітерації та ознаки успіху<br>
Мета: Запустити робочий цикл "Облік — Аналіз — Сповіщення" для однієї категорії товарів.<br>
Ознаки успіху<br>
Користувач витрачає менше ніж 2 хвилини на внесення даних про надходження товару.<br>
Система видає прогноз закупівлі з точністю +/- 10% від реальної потреби протягом тижня тестування.<br>
Головний сценарій (сповіщення про дефіцит) спрацьовує без технічних збоїв 7 днів поспіль.<br>
4. Карта стейкхолдерів (Розширена)<br>
Комірники (Користувачі): Очікують інтерфейс «в одну кнопку», що працює навіть у рукавичках або при поганому освітленні. Основне обмеження — мінімальний час на навчання.<br>
Власники бізнесу (Замовники): Очікують високу точність розпізнавання (мінімум помилок у цифрах). Головна вимога — швидкість окупності рішення.<br>
Команда розробки: Має забезпечити стабільну роботу ШІ-моделі OCR в умовах обмежених ресурсів та стислих термінів першої ітерації.<br>
5. Сценарій користувача (User Scenario)<br>
Умови початку: Користувач тримає в руках нову накладну, додаток відкрито на головному екрані.<br>
Крок 1: Натиснути велику іконку камери «Сканувати прихід».<br>
Крок 2: Сфотографувати документ так, щоб текст був чітким.<br>
Крок 3: Переглянути розпізнаний список назв та кількостей, що з'явився на екрані.<br>
Крок 4: Натиснути «Підтвердити та додати на склад».<br>
Очікуваний результат: Екран очищується, з'являється зелена позначка успіху, а товари миттєво відображаються у загальному реєстрі залишків.<br>
6. Критерії успіху та приймання (Definition of Success)<br>
Проєкт вважається успішним після першої ітерації, якщо:<br>
Граничний час: Весь сценарій (від фото до збереження) займає не більше 8 секунд.<br>
Автономність: Людина, яка бачить додаток вперше, виконує завдання без жодної усної чи письмової підказки.<br>
Стабільність: Під час тестування (10 завантажень поспіль) не виникає жодного критичного збою або аварійного завершення програми.<br>
7. Початковий перелік завдань (Backlog)<br>
UI Prototype: Створити макет головного екрана з фокусом на центральну кнопку дії.<br>
AI OCR Engine: Налаштувати базову модель розпізнавання тексту для виділення пар «назва товару — кількість».<br>
Data Flow: Розробити логіку миттєвого відображення розпізнаних даних для перевірки користувачем.<br>
Database Integration: Реалізувати функцію збереження підтверджених даних у глобальний список залишків.<br>
